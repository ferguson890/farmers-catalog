# Источник данных: чем хранить каталог

## Рекомендация
**Этап 1 (сейчас, до ~30–50 производителей): статический `data/farmers.json` в репозитории.**
**Этап 2 (когда наполнением займётся нетехнический сотрудник): Google Sheets → автоэкспорт в тот же `farmers.json`.**

Так мы получаем простоту и скорость статики сейчас и удобное наполнение «без кода» потом — без переписывания фронтенда (он в обоих случаях читает один и тот же JSON).

## Сравнение вариантов

| Критерий | JSON в репо | Google Sheets | Airtable |
|---|---|---|---|
| Стоимость | бесплатно | бесплатно | freemium, лимиты |
| Порог входа для редактора | нужен Git/PR | знаком всем (Excel-подобно) | средний |
| Скорость отдачи | мгновенно (статика) | через экспорт/кэш | через API (медленнее, лимиты) |
| Масштаб | до сотен записей | тысячи | тысячи + медиа |
| Медиа (фото) | файлы в репо/CDN | ссылки/Я.Диск | вложения «из коробки» |
| Риск стороннего сервиса | нет | средний (API/квоты) | выше (квоты, vendor lock) |
| Подходит для Tilda | да | да (через прокладку) | да (через прокладку) |

## Почему именно так

- **JSON сейчас** — данных всего 6, правит разработчик/Claude, нужна максимальная скорость и предсказуемость. Никаких внешних зависимостей и квот.
- **Google Sheets потом** — сотрудник Ассоциации заполняет таблицу (структура уже есть: лист «Карточки» в исходном реестре 1:1 ложится на нашу модель). Колонки = поля из `DATA-MODEL.md`.
- **Airtable** оправдан, только если понадобятся вложения-фото внутри базы, роли/права и сложные представления. Для текущей задачи избыточен и привязывает к вендору.

## Как подключить Google Sheets (этап 2)

Вариант А — **экспорт в JSON по расписанию (предпочтительно):**
1. Таблица с колонками = поля модели (`id`, `name`, `category`, …).
2. GitHub Action раз в N часов (или по кнопке) тянет Sheets через CSV-экспорт/Sheets API, валидирует, пишет `data/farmers.json`, коммитит.
3. Фронтенд не меняется — читает тот же файл. Сайт остаётся статикой (быстро, без рантайм-зависимости от Google).

Вариант Б — **чтение на лету:** опубликовать лист как CSV (`Файл → Опубликовать в интернете`) и в `catalog.js`/`farmer.js` заменить загрузчик на парсер CSV. Проще, но сайт зависит от доступности Google и медленнее.

> Рекомендуется **Вариант А**: контент удобно править в таблице, а пользователь получает быструю статику.

## Конвертация Excel-реестра в JSON (одноразово/повторяемо)
Структура реестра (`Реестр_Фермеров_Итог.xlsx`, лист «Карточки», 19 полей на фермера) уже отражена в нашей модели. Скрипт-конвертер (Python + pandas) можно положить в `scripts/xlsx_to_json.py` и переиспользовать при обновлении реестра. Маппинг полей — в `DATA-MODEL.md`.
